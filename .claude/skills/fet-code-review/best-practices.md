description: 評估「做得好的地方」的準則，基於 SOLID 與 Clean Code 原則
---

審查「做得好的地方」時，依以下準則尋找值得肯定的設計決策：

## SOLID 原則

- **S（單一職責）**：一個 function / class / composable 只做一件事
- **O（開放封閉）**：新增功能靠擴充而非修改既有程式碼（例如用 map 取代 switch）
- **L（里氏替換）**：子型別可安全替換父型別（TypeScript 介面繼承的正確使用）
- **I（介面隔離）**：型別定義精準，不強迫使用者依賴不需要的屬性
- **D（依賴反轉）**：依賴抽象（interface / type）而非具體實作

## Clean Code 原則

- **命名語意明確**：變數、函式、型別名稱能自我說明，不需要註解解釋
- **程式碼結構清晰**：邏輯分層合理，閱讀流程順暢，不需要反覆跳轉才能理解意圖
- **函式短小**：函式邏輯專注，不超過一個抽象層
- **避免魔術數字 / 字串**：使用常數或 enum 取代 hardcoded 值
- **提早回傳（Early Return）**：減少巢狀 if，讓主要邏輯保持在最淺層
- **不重複（DRY）**：共用邏輯抽取成 composable 或 utility
- **有處理邊界條件**：對 null、undefined、空陣列、未知值等邊界情況有明確處理，而非假設輸入必然合法

## TypeScript 特有

- **型別定義完整**：介面與型別覆蓋所有使用情境，不依賴隱式推斷或 `any`
- **型別安全**：善用 union type、narrowing、`as const`，讓錯誤在編譯期就被抓住
- **`satisfies` 運算子**：在保留推斷型別的同時做編譯期驗證
- **Readonly / 不可變性**：對不應被修改的資料使用 `readonly` 或 `as const`
